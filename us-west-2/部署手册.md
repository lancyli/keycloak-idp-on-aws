# Keycloak 高可用部署手册（AWS / CDK）

本手册说明如何使用 AWS CDK（TypeScript）在 **us-west-2（俄勒冈）** 区域部署一套生产级、高可用的 Keycloak 身份提供商（IdP），用于 Amazon Quick / QuickSight 单点登录（SSO）。全栈基于 ARM/Graviton。

> 单点登录的 Realm/客户端/SAML 联合配置属于部署后的一次性操作，见 [`USAGE-GUIDE.md`](./USAGE-GUIDE.md)。本手册只覆盖**基础设施部署**。
> AWS 中国（ZHY cn-northwest-1）的部署见同级 `../ZHY/` 目录，差异点见文末第 9 节。

---

## 1. 架构概览

```
 用户 (HTTPS)
   │
 CloudFront            强制跳转 HTTPS、禁用缓存、转发全部请求头/Cookie
   │  (HTTP 回源，且仅允许 CloudFront 托管前缀列表)
 ALB (多 AZ)           面向公网，安全组只放行 CloudFront origin-facing 前缀列表
   │  (HTTP 8080，私有子网)
 EKS Pod (Keycloak)    Graviton 节点 (AL2023 arm64)，2+ 副本，JDBC_PING 集群
   │  (5432，隔离子网)
 Aurora PostgreSQL     Graviton r7g，写实例 + 读实例，多 AZ，加密
```

**TLS 模型（暂未启用自定义域名时）：**

| 链路段 | 加密 | 控制方式 |
|--------|------|----------|
| 用户 → CloudFront | HTTPS（CloudFront 默认证书） | 对公网只暴露 HTTPS |
| CloudFront → ALB | HTTP（走 AWS 骨干网） | ALB 安全组**仅**放行 CloudFront origin-facing 前缀列表 |
| ALB → Pod | HTTP 8080（私有子网内） | 安全组锁定到 ALB 安全组 |

如需**端到端 HTTPS**，在 `lib/config.ts` 中设置 `cloudfront.customDomain.enabled = true` 并提供域名 + ACM 证书（见第 7 节），无需改动代码结构。

## 2. Stack 划分

| Stack | 内容 |
|-------|------|
| `keycloak-ha-network` | VPC（3 AZ）、public/private/isolated 三层子网、NAT 网关、VPC 端点 |
| `keycloak-ha-database` | Aurora PostgreSQL 16（Graviton）、写+读实例、Secrets Manager、安全组 |
| `keycloak-ha-eks` | EKS 集群、ARM 托管节点组、ALB Controller、Secrets Store CSI、ALB + 目标组，**以及 Keycloak 工作负载**（Deployment/Service/HPA/PDB/TargetGroupBinding + SecretProviderClass + 管理员密钥） |
| `keycloak-ha-cloudfront` | 位于 ALB 前的 CloudFront 分发 |

部署按依赖顺序：network → database → eks → cloudfront。

## 3. 默认资源规格（来自 `lib/config.ts`）

| 项 | 默认值 |
|----|--------|
| 区域 | us-west-2 |
| VPC CIDR | 10.60.0.0/16，3 AZ，3 个 NAT 网关 |
| EKS 版本 | 1.36（节点 m7g.large，min 3 / desired 3 / max 6，磁盘 30GiB） |
| Aurora | PostgreSQL 16.4，r7g.large，写实例 + 1 读实例，备份保留 14 天，开启删除保护 |
| Keycloak 镜像 | `quay.io/keycloak/keycloak:26.1.4`，命名空间 `keycloak` |
| Keycloak 副本/HPA | 2 副本，HPA 2~6，CPU 目标 70% |
| Pod 资源 | requests 500m/1Gi，limits 2000m/2Gi |

> 修改这些参数只需编辑 `lib/config.ts`，无需改 Stack 代码。

## 4. 前置条件

- 本地工具：Node.js 18+、AWS CLI v2、`kubectl`、Docker（CDK 打包资源需要）。
- 已为目标账号配置 AWS 凭据，并在 us-west-2 完成 CDK bootstrap：
  ```bash
  npx cdk bootstrap aws://<账号ID>/us-west-2
  ```
- 编辑 `lib/config.ts` 中的关键项：
  - **`eks.clusterAdminPrincipalArns`**（必填）：授予集群管理员权限的 IAM 角色/用户 ARN，即你运行 `kubectl` 所用的身份。**留空则部署后没有任何人能 `kubectl` 进入集群**（CDK 仍会部署工作负载，但无法手动运维）。支持 Role ARN（`arn:aws:iam::<账号>:role/...`）和 User ARN（`arn:aws:iam::<账号>:user/...`）。
  - **`eks.publicEndpointAllowedCidrs`**：把 EKS 公网 API 端点限制到你的出口 CIDR。留空 = 公网端点完全开放（**生产不推荐**）。

## 5. 部署前检查清单（在全新客户账号执行 `cdk deploy` 前）

- [ ] **工具就绪**：Node.js 18+、AWS CLI v2、`kubectl`、Docker 已运行。
- [ ] **凭据正确**：`aws sts get-caller-identity` 确认指向目标账号。
- [ ] **已 bootstrap**：`npx cdk bootstrap aws://<客户账号ID>/us-west-2`。
- [ ] **无残留缓存**：`cdk.context.json` 和 `cdk.out/` 不存在（它们会缓存上一个账号的 AZ/资源，synth 时自动重建，均已 gitignore）。
- [ ] **`lib/config.ts` 已填写**：
  - [ ] `eks.clusterAdminPrincipalArns` = 客户管理员 Role/User ARN（否则无 kubectl 权限）。
  - [ ] `eks.publicEndpointAllowedCidrs` = 客户办公/VPN 出口 CIDR（否则 EKS API 公开）。
  - [ ] `keycloak.publicUrl` = 首次部署留空（第二遍再固定，见第 6 节）。
  - [ ] （可选）按客户负载调整 Aurora / EKS 节点规格 / Keycloak 副本数。
- [ ] **区域与配额**：目标为 us-west-2，确认 EKS、Aurora、NAT/EIP 配额充足。
- [ ] **镜像可达**：账号 NAT 出口能拉取 `quay.io/keycloak/keycloak`（us-west-2 正常；中国区见第 9 节）。

## 6. 部署步骤

```bash
npm install
npm run build          # tsc 类型检查
npx cdk synth          # 渲染 CloudFormation
npx cdk deploy --all   # 约 25~35 分钟（EKS 和 Aurora 是耗时大头）
```

### 首次部署的两遍法（固定 KC_HOSTNAME）

第一遍 `cdk deploy --all` 时让 `keycloak.publicUrl` 保持为空。完成后，把输出
`keycloak-ha-cloudfront.KeycloakPublicUrl` 的值填入 `lib/config.ts` 的
`keycloak.publicUrl`，再执行：

```bash
npx cdk deploy keycloak-ha-eks
```

这样 Keycloak 才会生成正确的 `https://` 前端/issuer URL。
（若 `cloudfront.customDomain.enabled = true`，会自动使用自定义域名，无需这一步。）

### 部署后读取输出

- `keycloak-ha-cloudfront.KeycloakPublicUrl` → Keycloak 访问地址（如 `https://dxxxx.cloudfront.net`）。
- 管理员密码写入 Secrets Manager（`keycloak-ha/keycloak/admin`，用户名 `admin`）：
  ```bash
  aws secretsmanager get-secret-value --secret-id keycloak-ha/keycloak/admin \
    --region us-west-2 --query SecretString --output text
  ```

### 连接 kubectl 并验证

```bash
aws eks update-kubeconfig --name keycloak-ha --region us-west-2
kubectl -n keycloak get pods,svc,hpa,targetgroupbinding
```

正常情况下应看到 2 个 `keycloak-*` Pod 处于 `1/1 Running`、分布在不同节点上。

## 7. 升级为端到端 HTTPS（自定义域名）

1. 申请两张 ACM 证书：**us-east-1** 一张（CloudFront viewer 证书）、**us-west-2** 一张（ALB 监听证书）。
2. 在 `lib/config.ts` 设置：
   ```ts
   cloudfront: {
     customDomain: {
       enabled: true,
       domainName: 'idp.example.com',
       viewerCertArnUsEast1: 'arn:aws:acm:us-east-1:...:certificate/...',
       albCertArnUsWest2: 'arn:aws:acm:us-west-2:...:certificate/...',
     },
   },
   ```
3. `cdk deploy --all`，然后把 DNS 指向 CloudFront（CNAME `idp.example.com` → CloudFront 域名）。
   此时回源变为 HTTPS，`KC_HOSTNAME` 变为 `https://idp.example.com`。

## 8. 数据库连接说明（无需额外组件）

- 数据库是托管的 **Aurora PostgreSQL**，**不是集群内 Pod**，位于隔离子网。
- Keycloak Pod 通过 **JDBC 直连** Aurora 的 cluster writer endpoint（`KC_DB_URL=jdbc:postgresql://<writer>:5432/keycloak`），使用 Keycloak 自带连接池，**未部署 RDS Proxy/PgBouncer 等代理**。
- 数据库凭据由 Aurora 自动生成并存入 Secrets Manager，经 **Secrets Store CSI Driver + AWS Provider**（运行在 `kube-system`）同步成 k8s Secret `keycloak-secrets`，再由 Deployment 通过 `secretKeyRef` 注入；读取权限通过 **IRSA** 授予，全程无明文密码。
- 网络：EKS 集群安全组 → Aurora 安全组 5432 的入站规则；Aurora 侧 `rds.force_ssl=1` 强制 TLS。
- Keycloak 集群成员发现使用 `KC_CACHE_STACK=jdbc-ping`，复用同一数据库，**无需单独的服务发现组件**。

## 9. ZHY（cn-northwest-1）适配要点

本项目（us-west-2）**不可直接照搬到中国区**，需使用 `../ZHY/` 目录版本。主要差异：

- **分区**：所有 ARN 变为 `arn:aws-cn:...`（CDK 在 cn 区域会自动处理大部分）。
- **无 CloudFront**：中国区 CloudFront 需 **ICP 备案**，ZHY版以 **ALB + ACM** 作为公网入口（域名需备案）；无证书时仅 HTTP 测试。
- **ALB Controller 镜像**：必须从**中国区 ECR** 拉取（CDK 默认仓库在中国不可达），见 `config.eks.albControllerRepository`。
- **Keycloak 镜像**：`quay.io` 在中国不稳定，需先推送到 **cn-northwest-1 的 ECR** 并在 `config.keycloak.image` 引用。
- **机型/版本**：ZHY默认 m6g/r6g（Graviton2）、EKS 1.32（与 `KubectlV32Layer` 匹配）；上线前用 `aws eks describe-cluster-versions --region cn-northwest-1` 核实可用版本。
- **SAML 端点**：AWS 登录端点为 `signin.amazonaws.cn/saml`（而非 `.com`）。
- **DNS**：中国区无 Route 53，需在注册商/中国 DNS 服务商管理解析。

## 10. 清理

```bash
npx cdk destroy --all
```

注意：Aurora 开启了 `deletionProtection: true`，管理员/数据库密钥使用 `RETAIN` 策略——如需彻底清理，请手动删除这些资源。

## 11. 安全要点

- ALB 面向公网，但安全组只放行 CloudFront origin-facing 前缀列表。
- EKS 公网 API 端点应通过 `eks.publicEndpointAllowedCidrs` 限制来源。
- 数据库与管理员凭据存于 Secrets Manager，经 Secrets Store CSI（IRSA）注入，manifest 中无明文。
- Aurora 强制 TLS（`rds.force_ssl=1`）、静态加密、位于隔离子网。
- 目前**未启用 WAF**；上线前建议为 CloudFront 分发关联 `AWS::WAFv2::WebACL`。
