# Keycloak 高可用部署手册（AWS 中国 / ZHY cn-northwest-1）

独立的 CDK 项目。架构：**ALB（公网入口）→ EKS（Graviton）→ Aurora PostgreSQL（Graviton）**。**无 CloudFront**（中国区 CloudFront 需 ICP 备案）。全栈 ARM。

这是上级目录 us-west-2 部署的 AWS 中国（`aws-cn`）适配版。**不要**与 us-west-2 共用 `node_modules` / `cdk.out`；请在本 `ZHY/` 目录内部署。

> 单点登录的 Realm/客户端/SAML 联合配置见 [`使用指南.md`](./使用指南.md)（中文）或 [`USAGE-GUIDE.md`](./USAGE-GUIDE.md)（英文）。

---

## 1. 架构概览

```
 用户 (经 ALB ACM 证书走 HTTPS；测试时可走 HTTP)
   │
 ALB (多 AZ，面向公网)    安全组限制到 alb.allowedCidrs；有证书时 80->443 跳转
   │  (HTTP 8080，私有子网)
 EKS Pod (Keycloak)       Graviton 节点 (AL2023 arm64)，JDBC_PING 集群
   │  (5432，隔离子网)
 Aurora PostgreSQL        Graviton，写实例 + 读实例，多 AZ，加密
```

---

## 2. 与 us-west-2 版本的差异（中国区特性）

| 方面 | 适配方式 |
|------|----------|
| 分区（Partition） | `region = cn-northwest-1` 时，CDK 自动应用 `arn:aws-cn:*`。 |
| 公网入口 | **ALB**（非 CloudFront）。HTTPS 需要在**已 ICP 备案**的域名上配 ACM 证书（cn-northwest-1）（`alb.certArn` + `alb.domainName`）；否则仅 HTTP（测试用）。 |
| ALB Controller 镜像 | **必须**来自中国区 ECR —— `eks.albControllerRepository`。否则 CDK 默认仓库在 us-west-2 账号，中国不可达。**核实账号 ID** 及所选 controller 版本 tag 在该仓库中存在。 |
| Graviton 机型 | 默认 `m6g.large`（节点）与 `r6g.large`（Aurora）——在 cn-northwest-1 广泛可用。用 `m7g`/`r7g` 前请先核实。 |
| EKS 版本 | 默认 `1.32`。用 `aws eks describe-cluster-versions --region cn-northwest-1` **核实**，并保持 `@aws-cdk/lambda-layer-kubectl-vXX`（package.json）与之匹配。 |
| Keycloak 镜像 | `quay.io` 在中国不稳定——把镜像推送到 cn-northwest-1 的 ECR，并设置 `keycloak.image`。 |
| CSI driver / provider 镜像 | Helm 镜像来自 `registry.k8s.io` / `public.ecr.aws`，中国可能慢/被阻断。拉取失败时镜像到 ECR。 |
| Route 53 | 中国区不提供——在注册商/中国 DNS 服务商管理解析，把 ICP 域名指向 ALB。 |
| SAML 联合 | AWS 登录端点为 `https://signin.amazonaws.cn/saml`（稍后在 Keycloak 内配置）。 |

## 3. 前置条件

- AWS 中国账号（cn-northwest-1）凭据；完成 CDK bootstrap：
  ```bash
  npx cdk bootstrap aws://<中国账号ID>/cn-northwest-1
  ```
- 编辑 `lib/config.ts`：
  - `eks.albControllerRepository` —— 核实中国区 ECR 账号 ID。
  - `eks.clusterAdminPrincipalArns` —— **必填**：你的 `arn:aws-cn:iam::...` 管理员主体。支持 User ARN（`...:user/...`）和 Role ARN（`...:role/...`）。留空则部署后无人能 `kubectl` 进入集群。
  - `eks.publicEndpointAllowedCidrs` —— 限制 EKS 公网 API 端点。
  - `keycloak.image` —— 你的 ECR 镜像 URI。
  - `alb.allowedCidrs` / `alb.certArn` / `alb.domainName` —— 入口暴露范围与 TLS。

## 4. 部署前检查清单（AWS 中国）

在全新 AWS 中国账号执行 `cdk deploy` 前逐项检查。标注 **VERIFY** 的项无法在中国境外臆断——请先在 cn-northwest-1 控制台/CLI 确认。

- [ ] **工具就绪**：Node.js 18+、AWS CLI v2、`kubectl`、Docker 已运行（CDK 用 Docker 打包资源）。
- [ ] **凭据正确**：指向客户的 **AWS 中国** 账号（`aws sts get-caller-identity` 应返回 `arn:aws-cn:...` 身份）。
- [ ] **已 bootstrap**：`npx cdk bootstrap aws://<中国账号ID>/cn-northwest-1`。
- [ ] **无残留缓存**：`cdk.context.json` 和 `cdk.out/` 不存在（已 gitignore，synth 时重建）。
- [ ] **`lib/config.ts` 已填写**：
  - [ ] `eks.clusterAdminPrincipalArns` = 客户 `arn:aws-cn:iam::...` 管理员 User/Role（否则无 kubectl 权限）。
  - [ ] `eks.publicEndpointAllowedCidrs` = 客户出口 CIDR（否则 EKS API 公开）。
  - [ ] `eks.albControllerRepository` = 正确的中国区 ECR（ZHY默认 `961992271922`；北京 `918309763551`）。**VERIFY**。
  - [ ] `keycloak.image` = 客户 ECR 镜像 URI（quay.io 在中国不稳定）。**VERIFY** 镜像已推送。
  - [ ] `alb.allowedCidrs`（若内部使用，从 `0.0.0.0/0` 收紧）；如需 HTTPS，配 `alb.certArn` + `alb.domainName`（需 ICP 备案）。
  - [ ] `keycloak.publicUrl` = 首次部署留空（第二遍再固定——见下文）。
- [ ] **VERIFY** ALB Controller 镜像 tag（`v2.8.2`）在上述中国区 ECR 仓库中存在。
- [ ] **VERIFY** EKS 版本 `1.32` 可用：`aws eks describe-cluster-versions --region cn-northwest-1`（保持 `@aws-cdk/lambda-layer-kubectl-v32` 与之匹配）。
- [ ] **VERIFY** Aurora PostgreSQL `16.4` + `r6g.large`、节点 `m6g.large` 在 cn-northwest-1 可用。
- [ ] **VERIFY** 账号内 EKS、Aurora、NAT/EIP 配额充足。

## 5. 部署步骤

```bash
npm install
npm run build
npx cdk synth
npx cdk deploy --all   # network -> database -> eks
```

### HTTPS / KC_HOSTNAME（两阶段，与上级项目思路一致）

- **有 ICP 域名 + ACM 证书**：设置 `alb.certArn` + `alb.domainName`。`KC_HOSTNAME` 自动变为 `https://<域名>`，ALB 做 80→443 跳转。把 DNS 指向 ALB DNS 名称（Stack 输出 `AlbDnsName`）。
- **无域名测试**：保持 `alb.certArn` 为空（仅 HTTP）。部署后读取 `AlbDnsName`，设置 `keycloak.publicUrl = http://<alb-dns>` 并重新部署，使 Keycloak 生成正确 URL（避免管理控制台出现 "somethingWentWrong" / 加载错误）。

## 6. 部署后验证

```bash
aws eks update-kubeconfig --name keycloak-ha-cn --region cn-northwest-1
kubectl -n keycloak get pods,svc,targetgroupbinding,hpa
```

管理员密码：Secrets Manager `keycloak-ha-cn/keycloak/admin`。随后用 `signin.amazonaws.cn`
配置 `aws-realm`、OIDC/SAML 客户端以及 SAML 提供商/角色——精确的中国区（`arn:aws-cn`）步骤见
[`使用指南.md`](./使用指南.md)。

## 7. 数据库连接说明（无需额外组件）

- 数据库是托管的 **Aurora PostgreSQL**，**不是集群内 Pod**，位于隔离子网。
- Keycloak Pod 通过 **JDBC 直连** Aurora 写实例端点（`KC_DB_URL=jdbc:postgresql://<writer>:5432/keycloak`），使用 Keycloak 自带连接池，**未部署 RDS Proxy/PgBouncer 等代理**。
- 凭据由 Aurora 自动生成并存入 Secrets Manager，经 **Secrets Store CSI Driver + AWS Provider**（运行在 `kube-system`）同步成 k8s Secret `keycloak-secrets`，再由 Deployment 通过 `secretKeyRef` 注入；读取权限通过 **IRSA** 授予，全程无明文密码。
- 网络：EKS 集群安全组 → Aurora 安全组 5432 入站；Aurora 侧 `rds.force_ssl=1` 强制 TLS。
- 集群成员发现使用 `KC_CACHE_STACK=jdbc-ping`，复用同一数据库，**无需单独服务发现组件**。

## 8. 清理

```bash
npx cdk destroy --all
```

注意：Aurora 开启了 `deletionProtection: true`，管理员/数据库密钥使用 `RETAIN` 策略——如需彻底清理，请手动删除这些资源。

---

## 9. 安全说明

没有 CloudFront 时，ALB 直接面向公网，仅由 `alb.allowedCidrs` 限制（公开 IdP 默认 `0.0.0.0/0`——若内部使用应收紧到企业 CIDR）。生产环境建议为 ALB 关联 **AWS WAF**（中国区可用）。
