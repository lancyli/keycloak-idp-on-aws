# Keycloak 高可用部署手册（AWS ZHY，cn-northwest-1）

独立的 CDK 项目。架构：**ALB（公网入口）→ EKS（Graviton）→ Aurora PostgreSQL（Graviton）**。**无 CloudFront**（ZHY CloudFront 需 ICP 备案）。全栈 ARM。

这是上级目录 us-west-2 部署的 AWS ZHY（`aws-cn`）适配版。**不要**与 us-west-2 共用 `node_modules` / `cdk.out`；请在本 `ZHY/` 目录内部署。

> 单点登录的 Realm/客户端/SAML 联合配置见 [`使用指南.md`](./使用指南.md)（中文）或 [`USAGE-GUIDE.md`](./USAGE-GUIDE.md)（英文）。

---

## 1. 架构概览

```
 用户 (经 ALB ACM 证书走 HTTPS；测试时可走 HTTP)
   │
 ALB (多 AZ，面向公网)    安全组限制到 alb.allowedCidrs；有证书时 80->443 跳转
   │  (HTTP 8080，私有子网)
 EKS Pod (Keycloak)       Graviton 节点 (AL2023 arm64)，JDBC_PING 集群
   │  (5432，隔离子网)
 Aurora PostgreSQL        Graviton，写实例 + 读实例，多 AZ，加密
```

## 2. 与 us-west-2 版本的差异（ZHY特性）

| 方面 | 适配方式 |
|------|----------|
| 分区（Partition） | `region = cn-northwest-1` 时，CDK 自动应用 `arn:aws-cn:*`。 |
| 公网入口 | **ALB**（非 CloudFront）。HTTPS 需要在**已 ICP 备案**的域名上配 ACM 证书（cn-northwest-1）（`alb.certArn` + `alb.domainName`）；否则仅 HTTP（测试用）。 |
| AWS LB Controller | **不用**内置 `albController`（其 Helm chart 硬编码从 `aws.github.io/eks-charts` 拉取，ZHY不可达）。改为从**本地 chart 资产**（`charts/aws-load-balancer-controller/`，经 CDK 的 S3 下发，ZHY可达）安装 + 手动 IRSA ServiceAccount + IAM 策略（`lib/alb-iam-policy.json`，资源已改写为 `aws-cn` 分区）+ controller 镜像来自ZHY EKS ECR（`eks.albControllerRepository`，默认 `961992271922`）。 |
| Graviton 机型 | `m7g.large`（节点）与 `r7g.large`（Aurora）——Graviton3，已用 china profile 核实在 cn-northwest-1 可用（3 AZ），与 us-west-2 一致。 |
| EKS 版本 | `1.35`（与 us-west-2 一致）。cn-northwest-1 已支持到 1.36，但固定 `1.35` 以匹配可用的 `@aws-cdk/lambda-layer-kubectl-v35`（CDK 生态尚无 v36 包）。 |
| EKS 托管插件 | `vpc-cni`、`kube-proxy`、`coredns`、`metrics-server` 均以 **EKS 托管插件**安装（`resolveConflicts=OVERWRITE`），控制台 Add-ons 可见、可一键升级；`metrics-server` 支撑基于 CPU 的 HPA。ZHY托管插件镜像**自动**从区域 ECR（`961992271922`）拉取，无需额外配置。 |
| ALB Controller 版本 | `v2.17.1`（chart `1.17.1`，与 us-west-2 一致）。 |
| Secrets 注入 | **不用** Secrets Store CSI Driver（其 Helm chart 在 GitHub，ZHY不可达；且 CFN 动态引用在 eks manifest 里不会被解析）。DB/admin 凭据改由 **IRSA init 容器**（aws-cli 镜像）在 Pod 启动时从 Secrets Manager 拉取（**必须用区域 STS 端点**），写入内存卷（tmpfs），Keycloak 容器启动前 `export`。详见第 7 节。 |
| 容器镜像镜像化 | Keycloak（`quay.io`）和 aws-cli init 镜像（`public.ecr.aws`）在ZHY不稳/被阻断，需**镜像到本账号 cn-northwest-1 ECR**。`config.keycloak.image` 只填 `repo:tag`（如 `keycloak:26.6.4`）；完整 ECR 前缀（`<账号>.dkr.ecr.<区域>.amazonaws.com.cn`）在部署时按账号/区域**自动拼接**（源码不含账号 ID）。 |
| Route 53 | ZHY不提供——在注册商/ZHY DNS 服务商管理解析，把 ICP 域名指向 ALB。 |
| SAML 联合 | AWS 登录端点为 `https://signin.amazonaws.cn/saml`（稍后在 Keycloak 内配置）。 |

## 3. 前置条件

- AWS ZHY账号（cn-northwest-1）凭据；完成 CDK bootstrap（见第 5 节的环境变量）。
- **镜像准备（ZHY必做）**：把两个镜像镜像到**本账号 cn-northwest-1 ECR**（在一台能连 quay.io/public.ecr.aws 的机器上执行，如 Mac/区域外主机）：
  ```bash
  R=cn-northwest-1; ACCT=<你的ZHY账号ID>
  ECR="$ACCT.dkr.ecr.cn-northwest-1.amazonaws.com.cn"
  aws ecr create-repository --repository-name keycloak --region $R --profile china || true
  aws ecr create-repository --repository-name aws-cli  --region $R --profile china || true
  aws ecr get-login-password --region $R --profile china | docker login --username AWS --password-stdin $ECR
  # Keycloak（arm64，给 Graviton）
  docker pull --platform linux/arm64 quay.io/keycloak/keycloak:26.6.4
  docker tag quay.io/keycloak/keycloak:26.6.4 $ECR/keycloak:26.6.4 && docker push $ECR/keycloak:26.6.4
  # aws-cli(init 容器,arm64)
  docker pull --platform linux/arm64 public.ecr.aws/aws-cli/aws-cli:latest
  docker tag public.ecr.aws/aws-cli/aws-cli:latest $ECR/aws-cli:latest && docker push $ECR/aws-cli:latest
  ```
  > LB Controller 的**镜像**由ZHY EKS ECR（`961992271922`）自动提供，无需镜像化。
- **下载 LB Controller chart 到本地（ZHY必做；`charts/` 不入库）**：chart 不提交到仓库，部署前在能连 `aws.github.io` 的机器上拉取并解压到 `charts/`（CDK 会把它打包成 S3 资产下发，ZHY可达）：
  ```bash
  mkdir -p charts
  helm pull aws-load-balancer-controller --repo https://aws.github.io/eks-charts --version 1.17.1 -d charts/
  tar xzf charts/aws-load-balancer-controller-1.17.1.tgz -C charts/
  # 生成 charts/aws-load-balancer-controller/(eks-stack.ts 的 Asset 指向这个目录)
  ```
- 编辑 `lib/config.ts`：
  - `eks.clusterAdminPrincipalArns` —— **必填**：你的 `arn:aws-cn:iam::...` 管理员主体（User 或 Role ARN）。留空则部署后无人能 `kubectl`/控制台管理集群。
  - `eks.publicEndpointAllowedCidrs` —— 建议限制 EKS 公网 API 端点到你的出口 CIDR。
  - `eks.albControllerRepository` —— ZHY EKS ECR（ZHY 默认 `961992271922`；北京 `918309763551`）。
  - `keycloak.image` —— **只填 `repo:tag`**（默认 `keycloak:26.6.4`）；ECR 前缀自动按账号拼接，**勿写完整账号 URI**（避免账号 ID 入库）。
  - `alb.allowedCidrs` —— 允许访问 ALB 的来源 CIDR（默认 `[]` = 锁死，谁都进不来；填你的出口段，如公司/VPN 出口）。
  - `alb.certArn` / `alb.domainName` —— HTTPS 用（需 ICP 备案域名 + cn-northwest-1 ACM 证书）。

## 4. 部署前检查清单（AWS ZHY）

在全新 AWS ZHY账号执行 `cdk deploy` 前逐项检查。标注 **VERIFY** 的项无法在ZHY区域外臆断——请先在 cn-northwest-1 控制台/CLI 确认。

- [ ] **工具就绪**：Node.js 18+、AWS CLI v2、`kubectl`、Docker 已运行（CDK 用 Docker 打包资源）。
- [ ] **凭据正确**：指向客户的 **AWS ZHY** 账号（`aws sts get-caller-identity` 应返回 `arn:aws-cn:...` 身份）。
- [ ] **已 bootstrap**：`npx cdk bootstrap aws://<ZHY账号ID>/cn-northwest-1`。
- [ ] **无残留缓存**：`cdk.context.json` 和 `cdk.out/` 不存在（已 gitignore，synth 时重建）。
- [ ] **`lib/config.ts` 已填写**：
  - [ ] `eks.clusterAdminPrincipalArns` = 客户 `arn:aws-cn:iam::...` 管理员 User/Role（否则无 kubectl 权限）。
  - [ ] `eks.publicEndpointAllowedCidrs` = 客户出口 CIDR（否则 EKS API 公开）。
  - [ ] `eks.albControllerRepository` = 正确的ZHY ECR（ZHY默认 `961992271922`；北京 `918309763551`）。**VERIFY**。
  - [ ] `keycloak.image` = **`repo:tag`**（默认 `keycloak:26.6.4`，勿写完整账号 URI）。**VERIFY** 已镜像到本账号 ECR（见第 3 节）。
  - [ ] Keycloak 镜像 `keycloak:26.6.4` 与 aws-cli init 镜像 `aws-cli:latest` 均已推送到本账号 cn-northwest-1 ECR（arm64）。
  - [ ] `alb.allowedCidrs` = 你的出口 CIDR（默认 `[]` = 锁死无人可访问）；如需 HTTPS，配 `alb.certArn` + `alb.domainName`（需 ICP 备案）。
  - [ ] `keycloak.publicUrl` = 首次部署留空（第二遍再固定——见下文）。
- [ ] LB Controller chart 已下载解压到 `charts/aws-load-balancer-controller/`（见第 3 节，`charts/` 不入库）；其镜像 tag `v2.17.1` 在ZHY EKS ECR（`961992271922`）中存在。
- [ ] EKS 版本 `1.35`（已用 china profile 核实 cn-northwest-1 支持 1.30–**1.36**；固定 1.35 以匹配 `@aws-cdk/lambda-layer-kubectl-v35`）。
- [ ] Aurora PostgreSQL `16.4` + `r7g.large`、节点 `m7g.large` —— 已用 china profile 核实在 cn-northwest-1 可用（均覆盖 3 AZ）。
- [ ] **VERIFY** 账号内 EKS、Aurora、NAT/EIP 配额充足。

## 5. 部署步骤

> **一键脚本**：以上准备(镜像镜像化、拉 chart)与部署可用 `deploy-china.sh` 自动完成——
> `AWS_PROFILE=<你的ZHYprofile> ./deploy-china.sh all`(账号自动探测)。
> 也可分步:`prep`(仅镜像+拉chart)、`deploy`(仅构建+部署)、`clean-secret`(重试前清理残留密钥)。
> `pin-hostname`(可选):固定 `KC_HOSTNAME`——`PUBLIC_URL=https://<你的ICP域名> ./deploy-china.sh pin-hostname`;
> 不传 `PUBLIC_URL` 则用 ALB DNS 生成 `http://<alb-dns>`。HTTP 测试阶段可不做(主机名由转发头推导);
> 上 HTTPS/域名时用。⚠️ 该命令会重部署 eks 栈,会按 `alb.allowedCidrs` 对齐安全组,先把出口 CIDR 写进配置以免冲掉手动规则。
> 下面是脚本背后的手动等价步骤,供理解/排错。

**ZHY关键**：`cdk` 用 Node SDK 解析账号并调用 STS。默认走**全球 STS 端点**，不认 `aws-cn` 令牌 → 报 "Unable to resolve AWS account" / "no credentials"。必须用 `AWS_PROFILE` 环境变量方式 + **区域 STS 端点** + 显式账号/区域：

```bash
npm install
npm run build

# ZHY部署统一用这组环境变量(china = 你的ZHY profile)
export AWS_PROFILE=china
export AWS_REGION=cn-northwest-1 AWS_DEFAULT_REGION=cn-northwest-1
export AWS_STS_REGIONAL_ENDPOINTS=regional
export CDK_DEFAULT_ACCOUNT=<你的ZHY账号ID> CDK_DEFAULT_REGION=cn-northwest-1

npx cdk bootstrap aws://$CDK_DEFAULT_ACCOUNT/cn-northwest-1
npx cdk synth keycloak-ha-cn-eks
npx cdk deploy --all --require-approval never   # network -> database -> eks(约 20-25 分钟)
```

> - `--profile` 命令行标志在某些 SDK 代码路径(上下文查找)不生效,**务必用 `AWS_PROFILE` 环境变量**。
> - EKS 集群创建约 15-20 分钟,之后才是节点组/插件/LB Controller/Keycloak。
> - `sts get-caller-identity` 若不带 `--region` 会走全球端点报 `InvalidClientTokenId`,属正常;带 `--region cn-northwest-1` 即成功。

### 部署失败重试须知（RETAIN 密钥）

若 eks 栈创建失败回滚,`keycloak-ha-cn/keycloak/admin`(RETAIN 策略)会被保留,重试时会报 `already exists`。重部署前先删除:
```bash
aws secretsmanager delete-secret --secret-id keycloak-ha-cn/keycloak/admin \
  --force-delete-without-recovery --region cn-northwest-1 --profile china
```

### HTTPS / KC_HOSTNAME（两阶段，与上级项目思路一致）

- **有 ICP 域名 + ACM 证书**：设置 `alb.certArn` + `alb.domainName`。`KC_HOSTNAME` 自动变为 `https://<域名>`，ALB 做 80→443 跳转。把 DNS 指向 ALB DNS 名称（Stack 输出 `AlbDnsName`）。
- **无域名测试（仅 HTTP）**：保持 `alb.certArn` 为空。部署后读取 `AlbDnsName`，设置 `keycloak.publicUrl = http://<alb-dns>` 并重新部署，使 Keycloak 生成正确 URL。
  - 另外,Keycloak realm 默认 `sslRequired=external`,经 HTTP 从外部访问会返回 **"HTTPS required"**。**仅测试**时可在 pod 内放开(生产务必改回并上 HTTPS):
    ```bash
    POD=$(kubectl get pods -n keycloak -l app=keycloak -o jsonpath='{.items[0].metadata.name}')
    kubectl exec -n keycloak $POD -c keycloak -- sh -c \
      '/opt/keycloak/bin/kcadm.sh config credentials --server http://localhost:8080 --realm master --user admin --password "$(cat /creds/admin_pass)" && /opt/keycloak/bin/kcadm.sh update realms/master -s sslRequired=NONE'
    ```

## 7. 数据库连接说明（无需额外组件）

- 数据库是托管的 **Aurora PostgreSQL**，**不是集群内 Pod**，位于隔离子网。
- Keycloak Pod 通过 **JDBC 直连** Aurora 写实例端点（`KC_DB_URL=jdbc:postgresql://<writer>:5432/keycloak`），使用 Keycloak 自带连接池，**未部署 RDS Proxy/PgBouncer 等代理**。
- 凭据由 Aurora 自动生成并存入 Secrets Manager。**ZHY不用 Secrets Store CSI**（chart 在 GitHub，不可达）——改由 Pod 的 **IRSA init 容器（aws-cli）** 在启动时 `aws secretsmanager get-secret-value` 拉取 DB/admin 凭据（**用区域 STS 端点** `AWS_STS_REGIONAL_ENDPOINTS=regional`），写入内存卷（`emptyDir: Memory`，tmpfs），Keycloak 容器启动前 `export` 为环境变量。凭据**不进 etcd、不进 CFN 模板**；读取权限由 **IRSA**(SA `keycloak`)授予。
- 网络：EKS 集群安全组 → Aurora 安全组 5432 入站；Aurora 侧 `rds.force_ssl=1` 强制 TLS。
- 集群成员发现使用 `KC_CACHE_STACK=jdbc-ping`，复用同一数据库，**无需单独服务发现组件**。

## 8. 清理

```bash
npx cdk destroy --all
```

注意：Aurora 开启了 `deletionProtection: true`，管理员/数据库密钥使用 `RETAIN` 策略——如需彻底清理，请手动删除这些资源。

## 9. 安全说明

- **ALB 入口默认锁死**：`alb.allowedCidrs` 默认 `[]`，ALB 安全组无任何入站规则（谁都进不来）。请填入你的**出口 CIDR**（如公司/VPN 出口段）再部署；**切勿**用 `0.0.0.0/0`（会把 IdP 暴露给全网）。
- **凭据安全**：DB/admin 密码经 IRSA init 容器运行时注入内存卷，不进 etcd、不进 CFN 模板、不进代码仓库。
- **HTTPS**：生产必须走 HTTPS（ICP 域名 + cn-northwest-1 ACM 证书）。仅测试时才用 `sslRequired=NONE` 放开 HTTP，上线前改回 `external` 并配证书。
- 生产建议为 ALB 关联 **AWS WAF**（ZHY可用）。
